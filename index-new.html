This document describes the formal semantics of Shape Expressions (ShEx) schemas. The semantics is defined on a slight generalization of ShEx schemas, called the shapes language. The generalization consists in:

  • triple constraints use sets of predicates, instead of a single predicate;
  • value expressions are generalized to arbitrary Boolean combinations of constraints on the value of the focus node and constraints the neighbour triples of the focus node.
The generalization allows to have a simpler syntax, by seeing some of the ShEx constructs as syntactic abbreviations. This makes the description of the semantics shorter. After defining the shapes language and its semantics, we show how ShEx schemas translate to it.

RDF Graphs

Let IRI be the set of IRI, Lit be the set of literal values, and Blank be the set of blank nodes. Recall that an RDF graph is a set of triples in (IRI ∪ Blank) × IRI × (IRI ∪ Blank ∪ Lit). For a triple (s,p,o), s is called the subject, p is called the predicate, and o is called the object of the triple.

For a graph G, we define the following sets:

  • Nodes(G) is the set nodes of G and is defined as the set of IRI or blank nodes or literal values that occupy a subject or object position in some triple of G
  • Props(G) is the set of properties of G and is defined as the set of IRI that occupy a predicate position in some triple of G
  • InvProps(G) is the set of inverse properties of G and is defined as the properties of G decorated by a hat ^. That is, if prop is a property, then ^prop is an inverse property.
  • for a triple (node', prop, node) in G, its inverse triple is the triple (node, ^prop, node'). Note that inverse triples are note elements of the graph. [FOOTNOTE: inverse triples are going to be used for defining the semantics for inverse triple constraints]
  • neigh(G,n) is the neighbourhood of the node n in the graph G, and is defined as the set of all triples whose subject is the node n and all inverses of the triples whose object is n. More formally, neigh(G,n) = {(node, prop, node') | (node, prop, node') ∈ G} ∪ {(node, ^prop, node') | (node', prop, node) ∈ G}.

Additionally, we define

  • Prop as a synonym of the set IRI, that we are going to use specifically to designate IRI that appear as predicates in the triples in graphs
  • InvProp as the set inverse properties, that are IRIs decorated by the hat.

The Shapes Language

A shapes schema defines a set of named value constraints, and a set of named shapes. The names are used to refer to value constraints and shapes. The name of a value constraint is called value constraint label, and the name of a shape is called shape label.

Shapes Schema

Formally, a shapes schema Sch is given by a triple Sch = (VCLabels, SLabels, def) where

  • VCLabels is a set of value constraint labels,
  • SLabels is a set of shape labels, and
  • def is a definition function that with every value constraint label associates a value expression, and with every shape label associates a triple expression.

Typing of a Graph by a Shapes Schema

A typing is a set of node-shape associations of the form (node,S) where node is a node in a graph, and S is a shape label in a schema. A node-shape association intuitively indicates that the node node satisfies the constraint given by the definition of the shape S.

Formally, let Sch = (VCLabels, SLabels, def) be a shapes schema and G be a graph. A typing of G by Sch is a subset of Nodes(G) × SLabels.

Consider a shapes schema Sch = (VCLabels, SLabels, def) with SLabels = {<IssueShape>, <UserShape>, <EmployeeShape>}.

An example of a typing of the graph G by Sch is:

  • typing = { (inst:Issue1, <IssueShape>), (inst:User2, <UserShape>), (inst:User2, <EmployeeShape>) }

Value Expressions

Value expressions are associated with value constraint labels by the def function of a shapes schema. Value expressions are defined by the following abstract syntax.

ValueExpr ::= TripleExprRef
| NodeConstraint
| ConjunctiveValueExpr
| DisjunctiveValueExpr
| NegatedValueExpr
TripleExprRef ::= a shape label from SLabels
NodeConstraint ::= a subset of IRI ∪ Lit ∪ {blank}
ConjunctiveValueExpr ::= ValueExpr 'AND' ValueExpr
DisjunctiveValueExpr ::= ValueExpr 'OR' ValueExpr
NegatedValueExpr ::= 'NOT' ValueExpr

That is, a value expression is any Boolean combination of atomic components that can be:

  • a shape label (TripleExprRef) that refers to a triple expression defined in the shapes schema, or
  • a node constraint (NodeConstraint) that defines the set of allowed values for a node. In shapes schemas, a node constraint can be an arbitrary set that contains IRIs, literal values, or the special value blank that stands for any blank node. Triple Expressions

Triple expressions are associated with shape labels and are used as definitions for shapes. They are defined by the following abstract syntax.

TripleExpr ::= EmptyTripleExpr
| TripleConstraint
| SomeOfTripleExpr
| EachOfTripleExpr
| RepeatedTripleExpr
EmptyTripleExpr ::= 'EMPTY'
TripleConstraint ::= PropSet ValueExprRef
PropSet ::= a finite or co-finite subset of Prop ∪ InvProp
ValueExprRef ::= a value constraint label from VCLabels
SomeOfTripleExpr ::= TripleExpr '|' TripleExpr
EachOfTripleExpr ::= TripleExpr ',' TripleExpr
RepeatedTRipleExpr ::= TripleExpr '[' MinCard ';' MaxCard ']'
MinCard ::= a natural number
MaxCard ::= a natural number or '*'

That is, an atomic triple expression can be

  • the empty triple expression, that is only satisfied by a node which neighbourhood is empty, or
  • a triple constraint, that defines a constraint on a single triple. A triple constraint is composed of a set of properties or inverse properties, that define the allowed values for the predicate of a triple, as well as a reference to value constraint, that is to be satisfied by the node in the triple opposite to the focus node.

Then, a triple expression can be obtained by combining two triple expressions using the some-of binary operator, denoted |, and the each-of binary operator, denoted as a comma ,. Intuitively,

  • a some-of triple expression is satisfied if the neighbourhood of the node satisfies one of its sub-expressions, and
  • an each-of triple expression is satisfied if the neighbourhood of the focus node can be split in two parts, and each part satisfies one of the sub-expressions.
These however should not be considered as disjunction and conjunction, as we will see later on when the semantics will be formally defined.

Finally, a triple expression can be repeated by specifying a minimal cardinality that is a natural number, and a maximal cardinality that is a natural number of an unbounded value denoted as a star *. Intuitively, a repeated triple expression is satisfied if the neighbourhood of the focus node can be split into a number of parts that belongs to the interval specified by the minimal and maximal cardinality, and each of the parts satisfies the inner triple expression.

Parentheses are not part of the syntax of triple expressions, we use them here for delimiting sub-expressions. Note that, as we will see in the sequel, the some-of and each-of operators are associative and commutative, therefore we use them with any number of sub-expressions without parentheses. That is, for instance, (x | y) | z is the same as x | (y | z), so we write simply x | y | z, and similarly for the each-of operator.

Let a set of value constraint labels VCLabels = { <VC_A>, <VC_B>, <VC_string>, <VC_int> }

Let ex:p, ex:q, foaf:name, foaf:firstName, foaf:lastName be properties (that is, IRI), for some prefixes ex: and foaf:.

The following are examples of relatively simple triple expressions.

  • Triple constraints, with P = (Prop ∪ InvProp) \ {ex:p}
    • {^ex:p} <VC_A>
    • {foaf:name, foaf:lastName} <VC_string>
    • P <VC_int>
    The first two triple constraints use finite sets of properties, the last one uses a co-finite set of properties.
  • A repeated triple expression, which sub-expression is a triple constraint, with minimal cardinality 1 and maximal cardinality *
    • ( {ex:p} <VC_int> )[1;*]
  • An each-of operator on top of three triple constraints:
    • {ex:p} <VC_A> , {ex:q} <VC_int> , {ex:p} <VC_B>
  • A some-of operator on top of one triple constraint and one each-of sub-expression, the latter composed of one repeated triple constraint and one triple constraint.
    • {foaf:name} <VC_string> |
      ( ({foaf:firstName} <VC_string>)[1;*] , {foaf:lastName} <VC_string> )

We now give a more complex triple expression.

Let a set of value constraint labels VCLabels = { <VC_UserAndClient>, <VC_Programmer>, <VC_Tester>, <VC_User>, <VC_NotProgrammerTester>, <VC_AllNode> }

Let is:reproducedBy, is:reportedBy, is:affectedBy be properties for some prefix is:.

An example of a triple expression is (the leading numbers are line numbers for future reference): 

1.    {is:reportedBy}    <VC_UserAndClient>              ,
2.    {is:reproducedBy}  <VC_Tester>                     ,
3.    {is:reproducedBy}  <VC_Programmer>                 ,
4.  ( {^is:affectedBy}   <VC_User>                )[1;*] , 
5.    {is:reproducedBy}  <VC_NotProgrammerTester>        ,
6.  ( OtherProp          <VC_AllNode>             )[0;*]
where OtherProp = (Prop ∪ InvProp) \ { is:reportedBy, is:reproducedBy, ^is:affectedBy } is the set that contains all properties and inverse properties except for is:reportedBy, is:reproducedBy, ^is:affectedBy.

The above triple expression is an each-of expression on top of 6 sub-expressions, one on each line. The sub-expressions on lines 1., 2., 3., 5. are triple constraints. The sub-expression on line 4. is a repeted triple expression, with minimum cardinality 1 and maximum cardinality *. The sub-expression on line 6. is also a repeated triple expression, with minimum cardinality 0 and maximum cardinaliy *.

A triple expression defines a property on the neighbourhood of a node in a graph.

A set of triples Neigh is called a neighbourhood set of triples if there is a node such that all triple in Neigh is either of the form (node, prop, node') or of the form (node, ^prop, node') for some node' in G and some property prop or inverse property ^prop. Intuitively, a neighbourhood set of triples is a subset of neigh(G,node) for some node in the graph G.

For a neighbourhood set of triples Neigh from the graph G, a triple expression TExpr, and a typing of G by Sch, we say that Neigh satisfies TEXpr for typing, written typing, Neigh ⊢ TExpr, as defined above recursively on the structure of TExpr, where

  • P is a subset of Prop ∪ InvProp and V is a value constraint label (in rule te-tripleconstr);
  • ⊎ is the disjoint union of sets (in rules te-eachof and te-repet);
  • min and max are the minimum, respectively maximum cardinality in a RepetedTripleExpr (in rule te-repet).

A typing of G by Sch is called locally valid if for all node-shape association (node, S) in typing, it is the case that typing, neigh(node) ⊢ def(S).

Shapes Schemas with Stratified Negation

As we can see from the syntax, value expressions refer to shapes (through a TripleExprRef), and triple expressions refer to value expressions (through a ValueExprRef in a triple constraint). In order to avoid undesirable interaction between recursion and the negation in value expressions, we consider shapes schemas with stratified negation only, as defined in the sequel.

We start by some definitions that will allow us to define the stratification.

If VExpr is a value expression, and S is a shape label, we say that S appears negated in VExpr if S appears in VExpr under an odd number of negations, that is, under an odd number of occurrences of the NOT operator. We then define two dependency relations over the shape labels in a shapes schema, namely positive dependency dep+, and negative dependency dep-. Formally, for a shapes schema Sch = (VCLabels, SLabels, def), and two shape labels S and T in SLabels, we define:

  • dep+(S,T) if there exists a value constraint label V such that V appears in def(S), and T appears in def(V), but T does not appear negated in def(V). Recall that def(S) is a triple expression that defines the shape label S, and def(V) is a value constraint expression that defines the value constraint label V.
  • dep-(S,T) if there exists a value constraint label V such that V appears in def(S), and T appears negated in def(V).

Then the dependency relations dep+ and dep- define a dependency graph between shape labels of the shapes schema Sch. We say that the shapes schema Sch is with stratified negation if its dependency graph does not contain a cycle that contains a dep- edge.

It is well known that in case of stratified negation, one can define what is called a stratification. If the schema Sch = (VCLabels, SLabels, def) is with stratified negation, then there exists a natural number k and a partitioning function strat: SLabels → 1..k that with every shape label from SLabels associates a natural number between 1 and k, called the stratum of that label. Moreover, the partitioning function strat satisfies the following property: for all shape labels S,T in SLabels, if dep-(S,T), then strat(S) > strat(T), and if dep+(S,T), then strat(S) ≥ strat(T). Such a partitioning function is called a stratification. The property ensured of a stratification is that a shape label can depend negatively only on shape labels that are in strictly lower stratums, and a shape label can depend positively only on shape labels that are on lower or on the same stratum.

Shapes schemas with stratified negation are guaranteed to have well founded semantics as it will be defined in the sequel. It is not necessarily the case for shapes schemas that might contain cycles of negative dependencies. Therefore, a shapes schema is allowed only if it has stratified negation, and we consider only shapes schemas with stratified negation in the sequel. Remark that it is easy to check whether a schema is with stratified negation: it is enough to construct the dependency graph, and check that it does not contain any cycle that goes through a negative dependency.

Maximal Typing

Stratified schemas allow for the definition of a unique typing, which is the one used to determine whether a node in a graph satisfies a value constraint. This typing is called the maximal typing, as intuitively it forces to add in the typing every provable node-shape association.

The maximal typing is defined using a stratification. Let Sch = (VCLabels, SLabels, def) be a schema, strat: SLabels → 1..k be a stratification for Sch, and G be a graph. We define a sequence Sch1, ...., Schk of sub-schemas of Sch following the stratification strat, as follows. For all i in 1..k, Schi = (VCLabelsi, SLabelsi, defi) where SLabelsi is the subset of SLabels composed of the shape labels on the stratums 1..i, and VCLabelsi and defi are restricted on these labels only. Formally,

  • SLabelsi = strat-1(1) ∪ ... ∪ strat-1(i),
  • VCLabelsi are the value constraint labels that appear in def(S) for some S in SLabelsi, and
  • def1 is def restricted on SLabelsi and VCLabelsi only.
It follows from this definition that Schk is equal to Sch.

Then we define a sequence of typings typing1, ..., typingk such that for all i in 1..k, typing1 is the maximum typing for Schi. Formally,

  • typing1 is the union of all locally valid typings of G by Sch1;
  • for i in 2..k, typingi is the union of all locally valid typings of G by Schi that coincide with typingi-1 on the SLabelsi-1.

Note that because Sch is stratified and because of the way the sub-schemas Schi were defined, it is the case that all the typings typingi are locally valid.

The typing typingk as defined above is called the maximal typing of G by Sch.

Validation of a Graph against a ShEx Schema

Let Shex be a ShEx schema with set ShapeNames of shape names. For a graph G, a validation problem is defined as a set val of node-shape name associations. That is, val is a subset of Nodes(G) × ShapeNames. Then the validation problem consists in checking whether for every node-shape name association (node, sname) in val, it is the case that node satisfies the constraint defined by sname.

Formally, let Sch be the shapes schema that corresponds to Shex, and for every shape name sname in Shex, let valConstr(sname) be the corresponding value constraint name in Sch. Then the answer to the validation problem val is yes if and only if for all (node, sname) in val, it is the case that typing, node ⊢ valConstr(sname), where typing is the maximal typing of G by Sch.

Remark that the validation problem does not necessarily require to compute typing entirely. Only a portion of typing can be computed, starting from val and adding only the node-shape associations that are necessary for proving, or disproving, the node-shape name associations from val.
